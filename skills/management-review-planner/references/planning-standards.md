# Planning Standards

Use this note when planning a management or strategy literature review before prose drafting.

## Why Planning Comes First

A strong review usually fails or succeeds at the framing stage. If the high-level structure drifts during drafting, later evidence additions often produce repetition, concept slippage, and unstable research gaps. The planning stage therefore needs its own durable artifact.

## Core Rules

### 1. Freeze the architecture before drafting

The review should not begin as free-form prose generation. First build a persistent framework and revise it with the user. Once approved, the framework becomes the canonical outline.

### 2. Let the topic determine the constructs and relationships

Do not force every review into a one-concept template. Instead, decompose the user topic into the constructs, concepts, mechanisms, and relationships that the review must explain.

### 3. Trace definition anchors before relationship claims

Every construct used in the review should be defined carefully before the review claims how constructs relate to one another.

### 4. Make the framework specific enough for execution

The planner should not stop at section titles. It should specify what each section is meant to do and what each planned paragraph is supposed to contribute.

### 5. Archive every planning iteration

Each planning revision should create a timestamped snapshot under `07_plan/history/` so the user can inspect how the framework evolved.

## Minimum Contents of the Approved Plan

An approved plan should state:

- what the review is about
- which constructs, concepts, or mechanisms matter
- how they are defined and bounded
- what relationship, mechanism, or question the review explains
- how the review will be organized section by section
- what each planned paragraph is supposed to accomplish
- what contribution the final review aims to make

## Local Checklist

Before handing the project to the writing skill, verify that:

- `review_plan.md` exists
- the construct list and relationship logic have been confirmed
- the definition anchors have been identified
- the section blueprint is explicit
- the paragraph blueprint is explicit
- the user has confirmed the framework
- a timestamped archive snapshot exists for the latest planning revision
